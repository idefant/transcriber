# Мультиагентный процесс

Этот процесс нужен для задач, в которых одной локальной проверки недостаточно: новых функций, изменений UI, переиспользуемых компонентов, нетривиальных текстов, изменений нескольких экранов или задач с повышенным риском регрессии. Он не заменяет правила из `AGENTS.md` и документов `docs/agent/`.

## Выбор режима

Ведущий агент выбирает режим до начала работы и кратко объясняет выбор пользователю.

### Облегчённый режим

Работай без дополнительных агентов, если задача изолирована и её результат можно однозначно проверить локально. Примеры: изменение одного известного текста, небольшой безопасный рефакторинг в одном месте, ответ на вопрос о коде или обзор кода без внесения изменений.

Основной агент самостоятельно выполняет изменение или обзор, запускает релевантные проверки и сообщает результат. Если во время работы обнаруживается второй контекст использования, неоднозначность требований или риск регрессии, он переключает задачу на полный режим.

### Полный режим

Используй ведущего агента, агента-разработчика и агента-ревьюера, если задача меняет видимое поведение, затрагивает несколько сценариев, переиспользуемый компонент, тексты в нескольких контекстах, данные, навигацию, раскладку или если её результат нельзя безопасно вывести из краткого запроса.

Агенты работают последовательно. В общей рабочей директории нельзя одновременно поручать двум агентам изменять один и тот же набор файлов.

После передачи задания ведущий агент остаётся активным до получения отчёта исполнителя: он ожидает завершения, сам передаёт отчёт следующему агенту и без ожидания пользовательского сообщения запускает следующий обязательный этап. Статус «нет активных субагентов» не означает завершения полной задачи: ведущий агент сверяет, получены ли отчёты всех запланированных этапов и нет ли замечаний, требующих возврата разработчику.

## Согласование решения

Пользователь не обязан заранее писать полную спецификацию. До реализации ведущий агент исследует затронутую часть проекта и готовит пакет согласования на понятном пользователю языке:

- цель и предполагаемый пользовательский сценарий;
- экраны, места использования и состояния интерфейса;
- все предлагаемые тексты с указанием контекста;
- крайние случаи: загрузка, пустое состояние, ошибка, отключённое состояние и повторное использование компонента, если они применимы;
- критерии приёмки и способ проверки;
- явно отмеченные предположения и вопросы, на которые невозможно безопасно ответить по коду.

Для полной задачи, где решение, тексты или UX не были явно утверждены пользователем, ведущий агент останавливается на этом этапе и запрашивает подтверждение. После утверждения он передаёт задание разработчику. Явное поручение пользователя реализовать уже описанное решение считается подтверждением.

## Передача разработчику

Ведущий агент передаёт разработчику только проверяемое задание:

- утверждённую цель и критерии приёмки;
- границы изменения и известные затронутые файлы или области;
- все тексты и контексты их использования;
- обязательные сценарии ручной и визуальной проверки;
- ссылки на применимые правила из `AGENTS.md` и `docs/agent/`.

Разработчик не принимает продуктовую задачу самостоятельно и не расширяет её без возврата к ведущему агенту.

## Обязанности разработчика

До передачи результата на ревью разработчик обязан:

1. Реализовать только утверждённое решение и проверить каждый переданный сценарий.
2. Запустить `npm run format:check` и `npm run lint`. При нарушениях выполнить узконаправленное форматирование изменённых файлов или исправить код и повторить проверки.
3. Запустить все релевантные проверки из `docs/agent/after-work-checks.md` и дополнительные проверки, обязательные для затронутого типа изменений.
4. Для визуальных изменений выполнить процесс из `docs/agent/screenshot-testing.md`.
5. Передать ревьюеру отчёт: изменённые файлы, выполненные команды и их результаты, проверенные сценарии, принятые допущения и известные риски.

Если обязательная проверка не может быть выполнена или падает по причине вне изменения, разработчик не скрывает это: он фиксирует команду, результат и причину в отчёте.

Ведущий агент обязан передать этот отчёт ревьюеру вместе с утверждённым заданием; нельзя заменять отчёт кратким пересказом или оставлять ревьюера без сведений о проверках.

## Обязанности ревьюера

Ревьюер работает с исходным утверждённым заданием, diff и отчётом разработчика. Он не меняет код сам; исправления возвращаются разработчику.

Ревьюер проверяет:

- соответствие критериям приёмки и утверждённым текстам;
- все места использования изменённого компонента, пропса, текста или общего состояния;
- отсутствие статических текстов и предположений, пригодных лишь для одного контекста;
- состояния загрузки, ошибки, пустых данных и ограничения интерфейса, если они затронуты;
- соблюдение `AGENTS.md`, архитектурных правил и отсутствие лишних изменений;
- достоверность и достаточность отчёта о проверках.

Замечание должно содержать файл или область, наблюдаемую проблему, риск и ожидаемый результат. Ревьюер разделяет блокирующие замечания и необязательные предложения.

## Исправления и финальная приёмка

Разработчик исправляет блокирующие замечания, заново выполняет необходимые проверки и обновляет отчёт. Ревьюер повторно проверяет только исправления и их непосредственные последствия. Если после двух циклов сохраняется продуктовая неоднозначность, ведущий агент возвращает вопрос пользователю вместо нового предположения.

После одобрения ревьюера ведущий агент проводит независимую продуктовую приёмку: воспроизводит критерии приёмки в работающем приложении, проверяет все перечисленные контексты и повторно проверяет сценарии, затронутые исправлениями. Для UI он следует `docs/agent/screenshot-testing.md`. Только после этого ведущий агент сообщает пользователю о завершении задачи и о запущенных проверках.

Перед завершением ведущий агент также оценивает работу каждого привлечённого агента: сверяет его действия с назначенной ролью, утверждённым заданием и ограничениями проекта. В отчёте фиксируются нарушения роли, запрещённые действия и невыполненные обязанности либо явное отсутствие таких нарушений.
