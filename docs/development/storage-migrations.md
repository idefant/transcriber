# Версионирование хранилища и миграции

## Обзор

Приложение хранит данные в Tauri `app_data_dir` (`com.transcriber.desktop`). Конфигурация — настройки, провайдеры, конфигурация обработки и словарь — лежит в отдельных JSON-файлах. Постоянные данные в SQLite хранятся в базе `app.sqlite3`; пока это история диктовок. Аудиозаписи остаются WAV-файлами в подпапке `recordings`, а история хранит путь к файлу, а не сами байты.

Слой JSON-хранилища находится в `src-tauri/src/storage.rs`; доступ к базе истории — в `src-tauri/src/db.rs`; исполнитель миграций — в `src-tauri/src/migrations.rs`.

## Почему история в SQLite, а конфиги — в JSON

Ключевой критерий — модель доступа, а не формат. JSON хорош для мелких ограниченных по размеру конфигов: они всегда читаются целиком один раз и записываются целиком при изменении.

История же растёт без предела. Когда она лежала в одном `history.json`, каждая операция (сохранение диктовки, повтор, удаление, показ месяца) читала и переписывала **весь** файл. На ~900 записях / 7.5 МБ это стало достаточно тяжёлым, чтобы заморозить главный поток на ~350–600 мс в момент завершения диктовки. Заморозка выталкивала синтетический `Ctrl+V` вставки за окно восстановления буфера обмена (см. [clipboard-restore.md](clipboard-restore.md)): к моменту, когда WebView2 обрабатывал вставку, буфер уже был восстановлен, и вставлялось прежнее содержимое буфера, а не диктовка. Проявлялось это только при вставке в собственное окно приложения — именно тогда замороженный поток одновременно является и целью вставки.

SQLite убирает O(n) на записи: новая диктовка — это один `INSERT`, выборка за месяц — индексированный запрос, «последняя запись» — один запрос с `LIMIT 1`. Тяжёлые поля записи (снимок настроек, usage, детали ошибки) продолжают храниться как JSON, но внутри строки БД, поэтому парсятся только при открытии конкретной записи, а не при каждой операции.

### Схема истории

Таблица `history_records`:

- `id TEXT PRIMARY KEY`;
- `created_at TEXT` — UTC RFC3339, лексикографически сортируется как хронология (для сортировки и «последней записи»);
- `transcription_text TEXT` и `postprocessing_text TEXT` — тексты этапов для полнотекстового поиска;
- `data TEXT` — весь `HistoryRecord`, сериализованный в JSON. Колонки `id`/`created_at` и тексты этапов дублируют часть полей только ради запросов; при чтении запись восстанавливается из `data`.

Выборка за месяц вычисляет в приложении UTC-границы локального календарного месяца и фильтрует `created_at` полуоткрытым интервалом. Это сохраняет корректную группировку у границы месяца, не хранит производный месяц и использует индекс времени как для фильтрации, так и для сортировки.

Таблица `history_records_fts` — внешний FTS5-индекс двух текстовых колонок с trigram-tokenizer. Он ищет регистронезависимые подстроки длиной от трёх Unicode-символов; поиск по опечаткам намеренно не поддерживается. Триггеры синхронизируют индекс с основной таблицей при создании, изменении и удалении записи.

### Поиск по расшифровкам

`search_history_records` (`history.rs`) выполняет по этому индексу поиск по всей истории; SQL живёт в `db::search_data`.

Ограничения, которые нужно помнить при изменениях:

- **Минимум три символа** — не продуктовое решение, а свойство trigram-tokenizer: последовательности короче трёх символов он не индексирует. Команда отсекает такие запросы до обращения к базе.
- **Ввод оборачивается в фразу FTS5** (`build_fts_phrase_query`): внутри двойных кавычек `AND`, `OR`, `NOT`, `*`, `^`, `:` перестают быть синтаксисом и ищутся буквально, а сами кавычки экранируются удвоением. Без этого пользовательский текст интерпретировался бы как выражение FTS и падал бы на неожиданном символе.
- **Сортировка по `created_at DESC`, а не по релевантности** — так требует UI («от новых к старым»), и это позволяет переиспользовать индекс времени. Ранжирование `bm25()` здесь намеренно не применяется.
- **Пагинация по `LIMIT`/`OFFSET`** страницами по `SEARCH_PAGE_SIZE` записей: поиск не ограничен месяцем, поэтому без лимита выборка могла бы вернуть всю историю. Общее число совпадений считается отдельным `COUNT(*)` — фронтенду оно нужно для номеров страниц.

Команда `async` по той же причине, что и `get_history_groups`: десериализация страницы записей не должна выполняться в главном потоке.

### `get_history_groups` — асинхронная команда

Синхронные команды Tauri выполняются в главном потоке. Фронтенд перезагружает текущий месяц истории по **каждому** событию `history-updated` (в том числе во время диктовки, см. [state-management.md](state-management.md)), а выборка и десериализация целого месяца в главном потоке снова заморозила бы его и вернула ту же проблему со вставкой. Поэтому `get_history_groups` объявлена `async` — так она уходит в пул воркеров async-runtime, а не в главный поток. Блокировка соединения БД удерживается только на время SQL-выборки строк; десериализация записей происходит уже после снятия блокировки.

## Версионирование схемы

Лёгкий файл метаданных `_meta.json` отслеживает текущую версию схемы:

```json
{ "schemaVersion": 3 }
```

`migrations::run` вызывается один раз в `lib.rs::setup`, после инициализации базы истории (`db::init`, чтобы миграции могли писать в неё) и до чтения любых доменных хранилищ. Она проверяет `_meta.json`, выполняет все ожидающие шаги миграции, а затем перезаписывает `_meta.json` значением `CURRENT_SCHEMA_VERSION`.

### Определение первого запуска

Когда `_meta.json` отсутствует:

- **Доменные файлы существуют** → уже существующая установка, появившаяся до введения версионирования. Считайте её схемой v1 (миграция не нужна, просто проставьте файл метаданных).
- **Доменных файлов нет** → свежая установка. Сразу проставьте `CURRENT_SCHEMA_VERSION`, пропустив все шаги миграции.

### Миграция v3: импорт истории в SQLite

Шаг v3 переносит `history.json` в базу. Реализация — в `history::migrate_history_json_to_db` (там живёт тип записи). Все записи вставляются одной транзакцией. Исходный `history.json` не удаляется, а переименовывается в `history.pre-sqlite.bak`: даже если пользователь позже запустит более старую версию приложения, его данные не потеряны. Повреждённый `history.json` тоже сохраняется в резервную копию, а импорт продолжается с пустой базой, чтобы не блокировать запуск. Пустой файл просто удаляется.

## Защита от данных более новой версии

Стабильные и предрелизные (canary) сборки используют один и тот же `identifier`, а значит один каталог данных и один `_meta.json` (см. [release-pipeline.md](release-pipeline.md)). Из-за этого пользователь может открыть данными более новой версии более старой сборкой — например, попробовав canary и вернувшись на стабильную, либо переустановив старую сборку.

Если `schemaVersion` в `_meta.json` **больше** `CURRENT_SCHEMA_VERSION`, `migrations::run` возвращает `StartupData::TooNew` и **не понижает `_meta.json`**. Понижать версию нельзя: тогда старый код счёл бы данные более новой схемы своими и повредил бы их. `lib.rs::setup` в этом состоянии не запускает диктовку и оверлей, а фронтенд (`StartupGate`) показывает блокирующий экран (`DataTooNewScreen`) с предложением обновить приложение или сбросить данные.

Важное ограничение: эта защита есть только в тех сборках, куда она уже включена. Уже выпущенные более старые сборки её не имеют — задним числом их не защитить. Поэтому важно, чтобы схема оставалась **линейной**: релизы идут из одной ветки (canary — это тег с `-`, стабильная — без), поэтому один номер версии всегда означает одно и то же преобразование. Если когда-нибудь понадобится hotfix от старого стабильного тега с собственной миграцией, номера версий разойдутся и защита сломается — тогда нужен другой механизм (например, набор применённых миграций по id).

## Сброс данных приложения

`maintenance::reset_app_data` — аварийный выход из состояния «данные новее кода» (откатить версию приложения нельзя) и обычное действие в настройках. Он закрывает соединение с базой (на Windows открытый файл нельзя переименовать), переносит всё содержимое каталога данных в резервную подпапку `reset-backup-<timestamp>` и перезапускает приложение. Данные не удаляются — они остаются в резервной подпапке; после перезапуска миграции видят свежую установку. Перенос делается с несколькими повторными попытками: сразу после закрытия базы ОС может ещё мгновение держать файл занятым.

## Отказоустойчивая загрузка

Существует две функции загрузки JSON; какую использовать, зависит от того, насколько критичен тихий fallback для данного домена:

### `storage::load_json_or_default` — тихий fallback

1. Файл отсутствует или пуст → возвращается `T::default()`.
2. JSON успешно парсится → возвращается значение.
3. JSON не удаётся распарсить → файл переименовывается в `<name>.corrupt-<YYYYMMDD-HHMMSS>` как резервная копия, возвращается `T::default()`.

Используется для: `settings.json`, `processing.json`, `dictionary.json`, `_meta.json` — доменов, где значение по умолчанию является разумной отправной точкой и нет междоменных ссылок по ID, которые могли бы молча сломаться.

### `storage::load_json_strict` — резервная копия + ошибка при повреждении

1. Файл отсутствует или пуст → возвращается `T::default()` (например, свежая установка).
2. JSON успешно парсится → возвращается значение.
3. JSON не удаётся распарсить → файл переименовывается как резервная копия, возвращается `Err` с описательным сообщением.

Используется для `providers.json` — домена, где тихий откат к пустому значению по умолчанию вызвал бы каскадные сбои (повисшие ID провайдеров в конфигурации обработки).

Когда `load_json_strict` возвращает ошибку, она распространяется через команду Tauri как отклонённый Promise, а хранилище фронтенда показывает состояние ошибки в соответствующем разделе UI. Повреждённый файл сохраняется в резервной копии, чтобы пользователь мог восстановить свои данные.

> ⚠️ Всегда пишите шаг миграции вместо того, чтобы полагаться на fallback,
> когда схема домена меняется несовместимым образом.

## Атомарная запись

`storage::save_json` сначала пишет в `<name>.tmp`, затем выполняет `fs::rename` в целевой файл. `rename` в пределах одного тома файловой системы атомарен — сбой посреди записи не может оставить частично записанный файл. База истории использует режим WAL, поэтому запись не блокирует чтение и наоборот.

## Добавление шага миграции

1. Увеличьте `CURRENT_SCHEMA_VERSION` в `src-tauri/src/migrations.rs`.
2. Добавьте новую ветку в `run_migration_step`:

```rust
4 => migrate_to_v4(app),
```

3. Реализуйте `migrate_to_v4`:
   - Для JSON-доменов прочитайте старый файл, преобразуйте данные и запишите обратно через `storage::save_json`.
   - Для истории работайте с базой через `db` (при необходимости добавляя таблицы или колонки через `CREATE ... IF NOT EXISTS`).
   - Если чтение не удалось, либо верните ошибку (прерывает запуск), либо сделайте резервную копию и сброс — в зависимости от допустимой потери данных для этого домена.

Шаги миграции выполняются по порядку от `from_version + 1` до `CURRENT_SCHEMA_VERSION`. По возможности каждый шаг должен быть идемпотентным. Меняйте схему истории аддитивно (добавляйте таблицы/колонки, не переименовывая и не удаляя), чтобы более старый код по возможности переживал чтение более новой базы.
